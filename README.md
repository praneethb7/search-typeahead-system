# Search Typeahead System
A search-as-you-type suggestion system (like the autocomplete in search engines and
e-commerce sites). It suggests popular queries while you type, records submitted searches,
keeps query-popularity data up to date, and serves suggestions with **low latency** using a
**distributed cache** routed by **consistent hashing**.

> Built for the *System Design 101* assignment. The design deliberately exercises the course
> topics: consistent hashing, distributed caching, eviction & invalidation, CAP/PACELC,
> replication vs sharding, SQL vs NoSQL/LSM, and microservice-style API design.
> See [docs/HLD.md](docs/HLD.md) for the full high-level design.

---

## What it does

- **Typeahead suggestions** — `GET /suggest?q=<prefix>` returns up to 10 prefix matches, sorted
  by popularity (basic) or by a **recency-aware** score (enhanced).
- **Search submission** — `POST /search` returns `{"message": "Searched"}` and updates the
  query's count (via a batched writer, not a synchronous DB write).
- **Trending searches** — `GET /trending` returns queries that are hot *right now*.
- **Distributed cache** — suggestion results are cached across **3 Redis nodes**, with the owning
  node chosen by a **consistent-hash ring**. `GET /cache/debug?prefix=<p>` shows the routing.
- **Batch writes** — search submissions are buffered and aggregated, then flushed to SQLite in
  one transaction, drastically cutting DB writes.
- **Metrics** — `GET /metrics` and `scripts/benchmark.py` report p50/p95/p99 latency, cache hit
  rate, and the write reduction from batching.

## Architecture at a glance

```
Browser (debounced) ──► FastAPI ──► Consistent-hash ring ──► 3 Redis cache nodes
                            │                                     │ (miss)
                            │                                     ▼
                            │                              In-memory prefix trie (top-K)
                            ▼                                     ▲
                      Batch writer ──► SQLite (primary store) ────┘
                            ▲
                      Recency tracker ──► /trending
```

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| Backend | Python + **FastAPI** (uvicorn) | async, low-latency, auto Swagger docs at `/docs` |
| Frontend | **Vanilla HTML/CSS/JS** | no build step; served by FastAPI (single origin, no CORS) |
| Primary store | **SQLite** | file-based, ACID, real DB read/write counts for the batch metric |
| Cache | **3 × Redis** (Docker) | distributed cache routed by our own consistent-hash ring |

### UI / design
A bold editorial ("brutalist") theme — paper + black + red, heavy display type, hard borders and
hand-drawn marker accents. The page uses Google Fonts (Anton, Archivo Black, DM Serif Display,
Space Grotesk, Space Mono), so the first load needs internet; without it the browser falls back to
system fonts and the layout still works. All UI behaviour (debounced suggestions, dropdown,
keyboard navigation, trending, recency toggle, loading/error states) is in `frontend/app.js`.

## Setup

### 1. Prerequisites
- Python 3.10+
- Docker Desktop (for the 3 Redis cache nodes)

### 2. Start the cache nodes
```bash
docker compose up -d        # redis on 6379, 6380, 6381
docker compose ps           # confirm all 3 are healthy
```

### 3. Install Python dependencies
```bash
python -m venv .venv
# Windows PowerShell:
.venv\Scripts\Activate.ps1
# macOS/Linux:
# source .venv/bin/activate
pip install -r requirements.txt
```

### 4. Generate the dataset and load it
```bash
python scripts/generate_dataset.py     # writes data/queries.csv (>= 100,000 queries)
python scripts/ingest.py               # loads the CSV into SQLite (data/typeahead.db)
```
(Dataset details and source are in [the dataset section](#dataset) below.)

### 5. Run the app
```bash
uvicorn backend.main:app --reload
```
- UI:        http://localhost:8000/
- API docs:  http://localhost:8000/docs

## Dataset

The dataset is **generated** by `scripts/generate_dataset.py` from an open-source English
word-frequency list (bundled seed words). It combines head terms with realistic modifiers to
synthesize **≥ 100,000 distinct queries**, each with a popularity `count` drawn from a Zipf-like
distribution (so a few queries are very popular and a long tail is rare — like real search
traffic). Generation is **deterministic** (fixed random seed), so anyone can reproduce the exact
dataset. A small committed sample lives at `data/sample_queries.csv`; the full CSV is gitignored
and regenerated by the script.

Input format (CSV with header):
```
query,count
iphone,100000
iphone 15,85000
iphone charger,60000
java tutorial,40000
```

## API summary

| API | Purpose |
|---|---|
| `GET /suggest?q=<prefix>` | Up to 10 prefix-matching suggestions, sorted by score |
| `POST /search` | Submit a search; returns `{"message":"Searched"}` and records it |
| `GET /trending` | Currently trending searches (recency-aware) |
| `GET /cache/debug?prefix=<p>` | Which cache node owns the prefix + hit/miss |
| `GET /metrics` | Cache hit rate, DB read/write counts, latency percentiles |

Full request/response details: [docs/API.md](docs/API.md).

## Measuring performance

```bash
python scripts/benchmark.py
```
Fires a realistic mix of suggestion requests and prints p50/p95/p99 latency, cache hit rate, and
the DB-write reduction achieved by batching. Findings are summarized in
[docs/PERFORMANCE.md](docs/PERFORMANCE.md).
